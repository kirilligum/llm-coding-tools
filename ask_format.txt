You asked me clarifying questions. For each individual question, I need you to provide details, give options for thoughtful answers, and provide your recommendation.

Breaking this down further, for each question, I need you to:

Contextualize: Explain why you are asking the question. Define any new terms, provide examples, reference relevant pieces of code, and establish the context.

Present Options: Present 2–4 materially distinct options. Use exactly one option if the decision is obvious or architecturally mandated. Do not include options that are technically possible but architecturally inferior just to fill space.

Rank by Spectrum: Order the options from long-term benefits (robust, low-maintenance, scalable, minimizing tech debt) to short-term benefits (pragmatic, simple, immediate, low-risk, expedient, gets-things-done).

Follow Option Guidelines: Ensure the options adhere to the architectural guidelines and precedence hierarchy listed below.

Recommend: State which option you recommend and justify why.

Decision Precedence Hierarchy
When architectural principles conflict, resolve them strictly in this order:

Correctness and verified local context.

Security and data integrity.

Fit with existing architecture.

Simplicity and YAGNI (You Aren't Gonna Need It).

Observability and Fail-Fast mechanisms.

Performance (only where explicitly relevant and measured).

Extensibility (only when rigorously justified).

Guidelines for Options
Evidence Hierarchy: Base all logic and trade-offs strictly on this order of evidence: 1) Verified local source, 2) Existing architecture/tests, 3) Project schemas/config, 4) Official docs, 5) Known engineering principles. Explicitly label any assumptions.

The "One Way" Principle (YAGNI & KISS): Do not over-engineer. Propose exactly one obvious, direct way per option. Avoid redundant fallbacks, clever workarounds, or safety nets for highly unlikely edge cases.

DRY Principle: Avoid all logic, style, and code duplication. Utilize reusable components and pure functions.

SSoT Principle: Ensure every piece of state or core logic has exactly one centralized source to prevent synchronization bugs.

Functional Core, Imperative Shell: Strictly isolate side effects. Write core algorithmic logic as pure functions. Push I/O and state mutations to the absolute outer boundaries.

Dependency Inversion (External Boundaries Only): Apply abstraction strictly at external I/O, database, network, vendor, or queue boundaries. Do not wrap stable internal modules or pure functions just to satisfy an abstraction rule.

Design by Contract & Observability: Validate invariants immediately. Crash early and loudly on violations. Use structured logging (JSON/key=value) at external boundaries. Avoid unstructured print statements.

Explicit State Machines: Use explicit Finite State Machines (FSMs) or algebraic data types strictly for mutually exclusive lifecycle states or ordered workflows. Do not encode a single lifecycle as multiple booleans. (Independent facts like isAdmin or hasVerifiedEmail may remain simple booleans).

Mechanical Sympathy vs. Conciseness: Default to concise, clear code. Only micro-optimize when the path is measured, obviously asymptotic, or explicitly identified as a performance-critical hot path.

Transparent Semantics & Minimal Surface Area: Minimize cognitive load (Spartan local naming, descriptive global naming, strict indentation limits). Expose the absolute minimum API surface, defaulting everything to private/internal.

Verified API & Contextual Integrity: Fit the existing architecture. Never invent APIs, fields, or config keys. Rely only on verified local source, official docs, or schemas.
